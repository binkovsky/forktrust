# Templates and policy packs (v0.7.9)

`forktrust init` scaffolds a starter `.forktrustconfig` from a pre-baked
template so you do not have to read [config.md](./config.md) end to end
before your first worktree.

## Quick start

```bash
# Auto-detect from cwd files
forktrust init

# Or pick explicitly
forktrust init --template nextjs
forktrust init --template strict-ai
```

`init` refuses if `.forktrustconfig` already exists; pass `--force` to
overwrite or `--show` to print the chosen template to stdout instead.

## Shipped templates

```bash
forktrust template list
```

| Name | What it sets | Auto-detect signal |
|---|---|---|
| `minimal` | copy `.env.local` only | (never auto-detected) |
| `nextjs` | per-worktree port block + copy `.env*` + verify `npm run build` | `package.json` declares a `next` dep |
| `nextjs-strict` | nextjs + `[summary]` Conventional Commits contract | (never auto-detected) |
| `go-cli` | verify `go build / go test -race / go vet` + require_clean | `go.mod` present |
| `python-poetry` | verify `poetry check && poetry run pytest` | `pyproject.toml` has `[tool.poetry]` |
| `strict-ai` | policy pack: `[summary]` only, project-agnostic | (never auto-detected) |

Inspect any one before writing:

```bash
forktrust template show nextjs
forktrust template show strict-ai > .forktrustconfig.suggestion
```

## Auto-detect rules

`forktrust init` (without `--template`) picks the first match in this
order:

1. `package.json` exists AND contains a `"next"` dependency â†’ **nextjs**
2. `pyproject.toml` exists AND contains `[tool.poetry]` â†’ **python-poetry**
3. `go.mod` exists â†’ **go-cli**
4. Otherwise â†’ **minimal**

The JSON envelope reports `auto_detected: true` and a human-readable
`detection_reason` so agents can surface "why this template" to the user.

## Composing templates

The shipped templates intentionally do NOT compose by themselves â€” each
one is a complete `.forktrustconfig` you can edit. Two practical
patterns:

**Layer strict-ai on top of nextjs:**

```bash
forktrust init --template nextjs
forktrust template show strict-ai >> .forktrustconfig   # append [summary]
# Then dedupe by hand if both have any overlapping sections.
```

**Use strict-ai as the contract scaffold and add your build commands by hand:**

```bash
forktrust init --template strict-ai
# Now open .forktrustconfig and add your [verify] commands and [[hooks]].
```

## JSON output

```bash
forktrust init --json
forktrust template list --json
```

`init` returns:

```json
{
  "template": "nextjs",
  "path": "/abs/path/.forktrustconfig",
  "wrote": true,
  "overwrote": false,
  "auto_detected": true,
  "detection_reason": "package.json declares a \"next\" dependency"
}
```

`template list` returns an array of `{name, description, detect}`.

## MCP

Two MCP tools wrap these commands for AI agents:

- `forktrust_init` â€” same args as the CLI (`template`, `force`, `show`)
- `forktrust_template_list` â€” no args; returns the JSON array

The `forktrust_init` tool is designed so agents can scaffold a config
without escaping shell quotes (the original motivation for MCP). See
[mcp.md](./mcp.md).

## Safety

Templates are pure data â€” Go string constants â€” and `init` only writes
`.forktrustconfig` in cwd. It does NOT:

- Touch other files in your repo
- Modify `.gitignore`
- Run any commands
- Read network

The trust gate (`[[hooks.post_create]]` with `type = "command"`) still
applies after you init. None of the shipped templates contain command
hooks; if you add one yourself, `forktrust doctor` will tell you to
`forktrust trust` before they execute.

## Verification

Every shipped template is unit-tested:

- Parses as TOML
- Passes `RepoConfig.Validate()` (regex compiles, length bounds sane,
  hook fields complete)
- Starts with `# Generated by: forktrust init --template <name>` header

Adding a broken template breaks the `TestAllTemplatesAreValid` suite in
`internal/templates/`.
